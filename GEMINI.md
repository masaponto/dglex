# Gemini CLI Guidelines

出力は必ず日本語で行ってください。

## 開発フローの強制
1. **実装計画（Planning）の提示**:
   - 実装を伴う指示を受けた際、着手前に必ず「実装計画」を提示し、ユーザーの承認を得ること。
   - 計画には「変更の目的」「影響ファイル」「変更内容の要約」「テスト・検証戦略」を含めること。
   - 複数の実装アプローチがある場合は、トレードオフ（メリット・デメリット）を提示すること。
2. **Future Work の管理**:
   - 改善点を見つけたが今回のスコープ外とする事項は、必ず `FUTURE_WORK.md` に記録し、優先順位を整理すること。
3. **承認後の実行**:
   - ユーザーが「実行してください」等の承認を出した後にのみ、ファイル操作を開始すること。

## 技術的標準 (Engineering Standards)
1. **型安全性の徹底 (Typing)**:
   - `Any` の使用を避け、`typing` モジュールを活用した厳密な型定義を行うこと。
   - 複雑な設定項目には `dataclass` を導入して構造化すること。
2. **パフォーマンス (Lazy Import)**:
   - CLI ツールにおいては、`dgl`, `torch`, `matplotlib` 等の重いライブラリのインポートを、実際に必要になるまで遅延（関数内インポート等）させること。
3. **スケーラビリティと安全性**:
   - 描画や重い計算の前には必ずデータのサイズチェック（ノード数・エッジ数等）を行い、必要に応じてサンプリング、警告、または制限を行うロジックを含めること。
4. **ドキュメント (Docstrings)**:
   - 公開・非公開に関わらず、すべての関数に Google スタイルの Docstrings（Args, Returns, Raises）を記述すること。
5. **DGL/NetworkX の作法**:
   - DGL の公式ドキュメントに従い、NetworkX への変換や可視化の整合性を維持すること。

## テスト・検証戦略
1. **DRY なテスト**:
   - テストの重複を避け、`pytest.mark.parametrize` と `ids` を活用して効率的かつ分かりやすいテストを記述すること。
2. **徹底した検証**:
   - `uv run pytest` を実行し、既存機能のデグレードがないことを確認すること。
   - 必要に応じて手動検証用のスクリプト（例: `large_graph.bin` の生成など）も計画に含めること。
